RULES OF THE GAME

RULES OF THE GAME

rules

[ TECHNICAL CHOICES ]

The technical choices are made and assumed by the Feature Team.

rules

[ TECHNICAL CHOICES ]

The Feature Team must act responsibly to identify the choices that impact it exclusively and the choices that impact the organization.

The choices that exceed the scope of the Feature Team (eg, license, infrequent programming language) must be validated by the organization or by the peer convergence process .

rules

[ GOOD USE ]

The right tool for good use is a source of savings.

rules

[ GOOD USE ]

A bad tool imposed on everyone is a risk. The misuse of a good tool can have very damaging consequences. For example, poorly used Agile methods are dangerous.

tools must be questioned.

Excel is often a rational choice but it is not a tool to do everything ** (CRM, ERP, Datamart, …)

rules

[ BUILD VS. BUY ]

Privilege Build for the core business.

Consider Buy for the rest, case by case.

rules

[ BUILD VS. BUY ]

The more a tool carries a feature differentiating feature for the organization, the more it is intended to be built. The core business must allow specificity and adapt quickly and often. Some software packages are sometimes adapted to this need.

For the rest : SaaS, Open Source, Build or Owner are to be studied case by case.

rules

[ OPEN SOURCE ]

Make the most of Open Source.

Alternative choices must be supported.

rules

[ OPEN SOURCE ]

The proprietary solutions are a risk for the organization that must be able to resume maintenance if needed.

There are few proprietary tools that do not have open source alternatives.

The organization benefits from the Open Source Community and can pay back its contributions.

rules

[ MICRO-SERVICES ]

Develop stand-alone and weakly coupled services.

rules

[ MICRO-SERVICES ]

weak coupling must be the norm.

Each micro-service has a clearly defined interface.

This interface determines the link between the micro-services.

Domain Driven Design allows, especially with the Bounded Contexts , to anticipate this problem.

rules

[ DATA ]

Each service has its own ** data storage system.

rules

[ DATA ]

A Data Store is intended to be coupled only with a single micro-service.

The access to data from one micro-service to another is done exclusively via its interface.

This design implies consistency over time across the platform. It must be apprehended at all levels , including UX.

rules

[ PERIMETER ]

Each micro-service must have a reasonable functional perimeter, which "fits in the head".

rules

[ PERIMETER ]

A micro-service offers a reasonable number of features.

Do not hesitate to cut a micro-service when it begins to grow.

A service of reasonable size makes it possible to consider serenely the rewriting , if the need arises.

rules

[ REAGENT ]

The Reactive Manifesto opens the way towards the design of reactive architectures.

rules

[ REAGENT ]

Responsive programming focuses on the flow of data and the propagation of change. It is based on the pattern " Observer " contrary to the approach " Iterator ", more traditional.

The Reactive Manifesto sets the fundamental axes: availability and speed, resilience to breakdowns, flexibility , elasticity and message orientation.

rules

[ ASYNC-FIRST ]

asynchronous processes favor decoupling and scalability in favor of performance.

rules

[ ASYNC-FIRST ]

Exchange between applications must be asynchronous first.

Asynchronous exchanges naturally allow weak coupling, isolation and flow control ( back-pressure **).

synchronous communication should only be considered when the action requires it.

rules

[ EVENTS ]

The information system must be oriented events.

rules

[ EVENTS ]

The **** event-driven *functional* processes are naturally **** asynchronously ** implemented.

The event orientation allows to favor the implementation of approaches such as C ommand Q uery R esponsibility S egregation (CQRS) and Event Sourcing.

rules

[ MESSAGE BROKER ]

Privilege a simple, robust and powerful message-broker to a "smart pipe".

rules

[ MESSAGE BROKER ]

ESB showed limits : scalable maintenance is critical , both from a technical and organizational point of view.

**** Broker *messages like* Kafka offer a simple , durable and resilient solution.

Smart endpoints and Simple pipes is an architecture that works at scale: it's Internet.

rules

[ SYNCHRONIZATION ]

The full synchronization of the system should be thought of as soon as it is designed.

rules

[ SYNCHRONIZATION ]

If the synchronization between two systems is ensured by a event flow , the total resynchronization of these systems must be planned at design time.

An automatic **** *synchronization audit* (example: by samples) allows to measure and detect any possible synchronization errors.

rules

[ CENTRALIZATION ]

The configuration of the services is centralized , their discovery is ensured by a directory.

rules

[ CENTRALIZATION ]

The configuration of the micro-services is centralized for all environments.

A central directory ensures dynamic discovery of micro-services.

The **** global *scalability* depends on this directory.

rules

[ SANDBOX ]

Feature Teams provide a sandboxed environment.

rules

[ SANDBOX ]

Feature Teams maintain a sandbox environment (current version and upcoming release) to allow other teams to scale up.

In some non-nominal cases, features may be disabled in the development environment.

rules

[ DESIGN FOR FAILURE ]

Your system will crash!

Design it so that it is tolerant.

rules

[ DESIGN FOR FAILURE ]

Your system will fail , it's inevitable. It must be designed for this (Design For Failure).

Predict redundancy at all levels: hardware (network, disk, etc …), applications (multiple instances of applications), geographical zones, providers (example: AWS + OVH).

rules

[ TOOLKITS ]

Provide toolkits , do not impose strict frameworks.

rules

[ TOOLKITS ]

Attention to the technical components houses and transverse ! They are restrictive, expensive and difficult to maintain.

accelerators , toolkits , technical stacks can be pooled , free Feature Teams, avoiding a dogmatic approach.

rules

[ CLOUD ]

Public, private or hybrid, the cloud (IaaS or PaaS) is the standard for production.

rules

[ CLOUD ]

PaaS services are preferred , simple , and scale quickly.

IaaS services allow you to address cases requiring greater flexibility but require more operational work.

A private cloud is not a traditional virtualization environment, it relies on commodity hardware.

rules

[ INFRASTRUCTURE ]

Feature Teams do not manage the infrastructure, it is provided and maintained by the organization.

rules

[ INFRASTRUCTURE ]

Infrastructure issues are not within Feature Teams. Infrastructure must be provided and maintained by a cross-functional service.

rules

[ CONTAINERS ]

Containers provide the flexibility needed for heterogeneous tooling.

rules

[ CONTAINERS ]

Containers provide the flexibility needed by Feature Teams to enable heterogeneous tooling in a homogeneous context.

rules

[ ENVIRONMENTS ]

The use of containers makes it possible to overcome the problems of technical environments.

rules

[ ENVIRONMENTS ]

The containers (example: Docker) make it possible to be freed of the differences of environment.

The deployment process must be agnostic to the environment.

Some components such as databases should not be deployed in containers. Their deployment is still automated.

rules

[ METRIC ]

Measures must be centralized and accessible to all.

rules

[ METRIC ]

The metrics are accessible to everyone with different levels of granularity: detailed view for the relevant Team Feature, aggregations for other members of the organization.

Access to metrics does not imply access to unit data , it must be controlled to maintain confidentiality.

All environments are affected.

rules

[ QUALITY ]

software quality is a key factor.

rules

[ QUALITY ]

code reviews are systematic. They are conducted by members of the Feature Team or other members of the organization, as part of Continuous Improvement.

That is not you being audited but your code : "You are not your code!".

The qualimetry can be partly automated, but nothing beats the "new eye".

rules

[ AUTOMATED TESTING ]

automated testing is a non-negotiable prerequisite for continuous deployment.

rules

[ AUTOMATED TESTING ]

Automated testing ensures quality of the product over time.

It is a prerequisite to continuous deployment, it allows **** changes *and* frequent deployments **.

The production rollout becomes an anecdotal event!

rules

[ TEST LEVELS ]

Tests at all levels : unit, integration, functional, resilience, performance.

rules

[ TEST LEVELS ]

The integration and functional tests are the most important, they guarantee the effective operation.

unit tests are suitable for development.

performance tests measure performance over time.

resilience tests help to anticipate failures.

rules

[ BLANKET ]

Cover is the primary objective indicator of test quality.

rules

[ BLANKET ]

The code coverage by the tests is a good metric of code quality.

This is a necessary condition but not sufficient , the coverage of a bad test strategy can be high without guaranteeing the good quality of the code.

rules

[ SECURITY ]

security is a process , it should not be treated in response to problems.

rules

[ SECURITY ]

security experts can be integrated directly into Feature Teams if needed.

security experts are available in the organization for audit , awareness and forward.
